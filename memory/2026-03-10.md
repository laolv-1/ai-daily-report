## 2026-03-10 07:22 GMT+8 - 官方 Chrome Extension Relay 真相查明

- 已基于本地官方文档与扩展源码完成 Browser Relay 的"只在 Options 页有反应"问题溯源，关键结论如下：
  - **不是** YouTube / 普通网页的 CSP 拦截导致"点了没反应"
  - **不是** 缺少 `Allow access to file URLs` 或 `Site access = On all sites` 这类权限开关
  - 真正的核心前提是：**本机 relay `http://127.0.0.1:18792` 必须 reachable**
- 官方文档证据：`/root/openclaw/docs/tools/chrome-extension.md`
  - 明确说明扩展架构为：browser control service + local relay server (`127.0.0.1:18792`) + Chrome MV3 extension
  - 明确写出 badge 语义：
    - `ON` = attached
    - `…` = connecting
    - `!` = **relay not reachable**
  - 明确说明：点击工具栏按钮才 attach；若看到 `!`，应去 Options 页查看 relay reachability
- 扩展源码证据：
  - `manifest.json` 权限仅为：`debugger`、`tabs`、`activeTab`、`storage`
  - `host_permissions` 仅包含：`http://127.0.0.1/*` 与 `http://localhost/*`
  - 由此可知其不是 content-script 注入模型，也没有官方要求必须开启"所有网站访问"或"允许文件网址"
- `background.js` 真相：
  - 扩展按钮点击后执行 `connectOrToggleForActiveTab()`
  - 首先调用 `ensureRelayConnection()` 对 `http://127.0.0.1:18792/` 做 `HEAD` 预检
  - 若 relay 不可达，则进入错误分支：
    - badge 设为 `!`
    - title 改为 relay not running
    - 自动执行 `openOptionsPage()`
- `options.js` 真相：
  - Options 页本质只是 relay 端口配置 + reachability 检查面板
  - 它会直接对 `http://127.0.0.1:<port>/` 发 `HEAD` 检查，并显示 `Relay reachable` 或 `Relay not reachable`
- Durable 结论更新：
  - "扩展只在 Options 页有反应"更接近**官方设计的失败兜底行为**，不是"普通网页权限不够"
  - 以后排障顺序必须固定为：
    1. 先查 `127.0.0.1:18792` relay 是否 reachable
    2. 再查扩展 badge (`ON` / `…` / `!`)
    3. 再查 tab 是否真的 attached
    4. 最后才讨论页面本身
  - 正式账号动作的战略主路仍然是：**`Chrome_龙虾出海`（Profile 2）+ 可视化 CDP (`9222`)**；官方 Relay 是"接管现有 tab"的平行能力，不应与 `9222` 肉身接管混为一谈。
