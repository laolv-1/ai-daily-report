## 2026-03-10 03:05 CST — Win10 官方插件接管法与移动端节点路线研究（只读审计）

- 已基于本地官方文档确认 Win10 浏览器官方正赛主路：**Chrome Extension Relay +（必要时）node host**，而不是继续依赖裸 `--remote-debugging-port=9222`。
- 官方 `chrome-extension` 文档关键结论：
  - 组成结构为 `browser control service` + 本地 `relay server`（默认 `http://127.0.0.1:18792`）+ Chrome MV3 扩展；
  - 扩展通过 `chrome.debugger` 附着用户**显式点击**的当前标签页；
  - OpenClaw 内建 `chrome` browser profile 直接用于该 relay 路线；
  - 若 Gateway 不在浏览器本机，官方推荐在浏览器机器上运行 **node host**，由 Gateway 代理浏览器操作；
  - 官方安全建议是 Gateway/node host 保持 tailnet-only，不暴露 relay/control 端口到公网。
- 已形成对“插件中转 vs 裸 CDP” 的 durable 认知：
  - 裸 CDP 方案依赖后台点亮整个 profile 的远程调试端口，易被 Win10 会话隔离、Profile 残留、Chrome 吞参等问题击穿；
  - 插件法是在用户已打开且已登录的真实 Chrome 标签页内，靠扩展按钮 attach 当前 tab，因此更能绕开旧式命令行夺舍在复杂 Win10 环境中的高摩擦问题；
  - 后续若目标是接管主公真实账号窗口，应优先研究并落地扩展 attach 流程，而不是继续把 `Profile 2 + 9222` 当战略主路。
- 已整理 Win10 官方插件接管 SOP（只读规划版）：
  1. `openclaw browser extension install`
  2. `openclaw browser extension path`
  3. 在 `chrome://extensions` 打开开发者模式并 `Load unpacked`
  4. pin 扩展到工具栏
  5. 在目标标签页点击扩展按钮，badge `ON` 表示 attach 成功
  6. 后续通过内建 `profile="chrome"` 进行控制
