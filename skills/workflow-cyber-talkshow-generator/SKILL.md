# Workflow Skeleton

按以下顺序执行整链：

1. 明确输入：主题、时长、语气、是否要毒舌、输出文件名。
2. 收集素材：优先读取本地情报/素材文件；若需外网信息，先抓取再汇总成结构化要点。
3. 生成文案：调用主模型 `lck/gpt-5.4` 输出适合 TTS 的短视频口播稿。
4. 清洗文案：去除不适合朗读的符号、超长句、表格、链接。
5. 调度 Win10 浏览器：只走官方 `chrome` relay 路线，不再走裸 CDP。
6. 打开免费 TTS 网页：优先选低门槛、少验证、支持直接下载音频的网站。
7. 自动注入文本：优先 DOM/浏览器控件；必要时切原生窗口控件输入。
8. 触发生成并等待下载：记录下载文件名、下载目录、完成时间。
9. 归档：落盘到约定目录，写入本地日志。
10. 回传：把最终 MP3/WAV 发给主公，附一句简短摘要。

## TTS 站点选择策略

优先级：

1. 无需登录
2. 无验证码/无复杂滑块
3. 允许直接输入中文
4. 允许直接下载 MP3/WAV
5. 页面结构稳定，便于 DOM 定位

首选候选：

- `ttsmaker.com`
- 其他无需登录、可直接生成下载的公开 TTS 页面

如主站风控升高，则切备用站；不要死磕一个站。

## 浏览器自动化约束

- 浏览器必须优先使用 Win10 官方 Chrome Extension Relay
- 使用 `openclaw browser --browser-profile chrome ...` 作为标准主路
- 仅在浏览器控件不可用时，才切原生窗口控件链

## 自我纠错规则

若浏览器或原生控制失败：

1. 记录错误症状（按钮没找到、下载未触发、焦点丢失、文本未注入等）
2. 先在本地做一次原因分类
3. 按同类问题的既有 SOP 自动重试
4. 最多重试 3 次
5. 仅在 3 次后仍失败时，向主公汇报“人类可读”的阻塞结论

不要把原始 Python traceback 直接丢给主公。

## 失败分流建议

- **DOM 选择器失效**：先 `snapshot` 重抓页面结构，再替换 ref/定位逻辑
- **浏览器控件不可点**：改走原生窗口聚焦 + 键盘输入/点击
- **下载无响应**：检查是否弹出二次确认、文件选择器或浏览器下载拦截
- **页面风控升级**：切换备用 TTS 站，不要长时间硬撞

## 输出纪律

对主公只输出：

- 当前跑到哪一步
- 是否成功拿到音频
- 若失败，给一句人类可读的阻塞结论

不要输出大段无意义调试噪音。
